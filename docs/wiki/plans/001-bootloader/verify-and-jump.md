# Verify-and-Jump (Plan 001 Phase 1.6)

This page documents the cryptographic gate added to the bootloader by
Phase 1.6. Phase 1.5's [bootloader-skeleton.md](bootloader-skeleton.md)
parses the slot-A `img_header_t` and jumps. Phase 1.6 inserts SHA-256 +
ECDSA-P256 verification between the parse and the jump. A failing verify
halts; slot-B fallback is Phase 1.7.

## Verify sequence

```
img_header_parse(SLOT_A_BASE) succeeds
   │
   ▼
DWT->CYCCNT = 0; enable cycle counter        (CoreDebug->DEMCR | TRCENA, DWT->CTRL | CYCCNTENA)
   │
   ▼
crypto_sha256(SLOT_A_BASE + payload_offset, payload_size) → computed[32]
   │
   ▼
crypto_memcmp_ct(computed, hdr.sha256, 32) == 0 ?
   │ no                                      ───►  "BL: verify FAILED: sha mismatch"  → halt
   │ yes
   ▼
crypto_ecdsa_p256_verify(bootloader_pubkey, computed, hdr.signature) == 1 ?
   │ no                                      ───►  "BL: verify FAILED: ecdsa reject"  → halt
   │ yes
   ▼
cycles = DWT->CYCCNT;  ms = cycles / 100000  (SYSCLK = 100 MHz)
"BL: verify ok in <cycles> cycles (~<ms> ms)"
   │
   ▼
jump_to_app(SLOT_A_BASE + payload_offset)
```

The constant-time SHA compare runs before ECDSA verify so a payload bit-flip
short-circuits the expensive curve operation. Even when the SHA matches,
ECDSA still runs against `hdr.signature`, so a header-clean tamper on the
signature itself is also caught.

## Log-line grammar

| Outcome                              | Line on USART2                                  |
|---|---|
| Header parse failed (Phase 1.5 path) | `BL: slot A header parse failed: rc=0xNNNNNNNN` |
| Image type wrong (Phase 1.5 path)    | `BL: slot A image_type != APP`                  |
| Payload SHA mismatches header SHA    | `BL: verify FAILED: sha mismatch`               |
| ECDSA signature rejected             | `BL: verify FAILED: ecdsa reject`               |
| Verify ok                            | `BL: verify ok in <cycles> cycles (~<ms> ms)`   |
| About to jump                        | `BL: jumping to slot A @ 0x080101NN`            |

`BL: verify FAILED:` is the prefix the HIL test
([scripts/run_verify_test.py](../../../../scripts/run_verify_test.py))
greps for; the suffix narrows the diagnosis.

## Timing budget

Plan 001 §1.3 caps verify time at **< 500 ms** on the F411 @ 100 MHz
(50 000 000 cycles). The bootloader brackets the SHA + memcmp + ECDSA
calls with the DWT cycle counter and emits the result on every boot.
[tests/baselines/bootloader_verify.json](../../../../tests/baselines/bootloader_verify.json)
records a soft baseline (50 % tolerance, since payload size varies per
build) and the hard cap. `run_verify_test.py` enforces the hard cap and
warns on soft regressions.

The first measured value is captured by running `run_verify_test.py` once
on the dev board; the baseline file is then updated in a follow-up
commit. Until that commit lands the soft check is skipped and only the
hard cap is enforced.

## Sector-0 budget

The bootloader plus `libcrypto.a` (SHA-256 ~1 KB, micro-ecc P-256 verify
with `uECC_SQUARE_FUNC=0` ~5 KB) must stay under 16 KB.
[apps/bootloader/loader/Makefile](../../../../apps/bootloader/loader/Makefile)
runs a post-link `stat`-based size guard against `loader.bin` (the
authoritative artifact, which includes the `.data` LMA copy that the
linker `ASSERT` does not see). Phase 1.6's measured size:

```
text     data     bss      total
10340    408      1904     12652  (loader.elf, before .data LMA)
                           10748  (loader.bin, after objcopy)
```

~5.6 KB headroom remains for Phase 1.8 OTA framing.

## Public key linkage

The bootloader links `bootloader_pubkey[64]` from `$(BL_PUBKEY_C)`,
generated by [tools/keygen.py](../../../../tools/keygen.py) from
`KEY_SEED = "stm32-bare-metal-dev-fixture"`. The signer
([tools/sign_image.py](../../../../tools/sign_image.py)) consumes the
matching private key from the same seed every build, so verifier and
signer cannot drift.

Production deployment would replace the deterministic seed with an
HSM-backed signing key and a one-time provisioning step that programs
the corresponding public key into a write-protected sector — out of
scope for the F411 dev project per Plan 001 §"Out of scope".

## Changes from Phase 1.5

- [apps/bootloader/loader/main.c](../../../../apps/bootloader/loader/main.c) —
  added DWT init, `crypto_sha256`/`crypto_memcmp_ct`/`crypto_ecdsa_p256_verify`
  calls, two distinct failure log lines, and the `verify ok in <cycles>`
  success log line. New `uart_print_dec32` helper.
- [apps/bootloader/loader/Makefile](../../../../apps/bootloader/loader/Makefile) —
  links `$(CRYPTO_LIB)` ahead of `$(IMG_LIB)`, adds `-I lib/crypto/inc`,
  and enforces a 16 384-byte hard cap on `loader.bin` after objcopy.
- [Makefile.common](../../../../Makefile.common) — exports
  `CRYPTO_LIB := $(BUILD_DIR)/lib/crypto/libcrypto.a` so future apps can
  link the same library without re-deriving the path.
- [scripts/run_verify_test.py](../../../../scripts/run_verify_test.py) —
  new HIL test: clean signed image must boot; payload-tampered image
  must be rejected and the app must never run.
- [.github/workflows/ci.yml](../../../../.github/workflows/ci.yml) —
  new `Run verify test` step in the `hil-tests` job, after
  `Run boot smoke test`.
- [tests/baselines/bootloader_verify.json](../../../../tests/baselines/bootloader_verify.json) —
  new file; baseline captured on first dev-board HIL run.
